Entendendo Classe e Objeto

O que são, como se relacionam e por que são a base de tudo em POO.

Parte 1 · 1.1

O que é uma Classe?

Uma classe é um molde, receita ou planta para criar objetos de um tipo específico. Ela define:

  • Quais dados (informações) um objeto desse tipo terá.
  • Quais ações (comportamentos) um objeto desse tipo pode realizar.
Analogia no mundo real — A Receita do Bolo

Pense em uma classe como a receita de um bolo:
• A receita é a classe. Não é um bolo de verdade; é um manual que diz como fazer um bolo.
• Um bolo feito a partir da receita é um objeto.
• Se você fizer o mesmo bolo 3 vezes, tem 3 objetos diferentes (cada bolo é uma instância).
Parte 1 · 1.2

O que é um Objeto?

Um objeto é uma instância concreta criada a partir de uma classe. É quando você realmente faz o bolo.

Exemplo – Sistema de Hospital

Classe: Paciente

  • Define: "Todo paciente tem um nome, uma idade e um convênio."
  • Define: "Todo paciente pode marcar uma consulta, atualizar seu telefone."

Objetos (instâncias da classe Paciente):

Paciente João, 30 anos, tem convênio Paciente Maria, 45 anos, sem convênio Paciente Carlos, 62 anos, tem convênio

Cada um é um objeto diferente, mas todos seguem o molde da classe Paciente.

Parte 2 · 2.1

Atributos: o que o Objeto "TEM"

Atributos são as características ou propriedades que um objeto possui. São os dados que descrevem aquele objeto.

Paciente · Hospital
  • nome
  • idade
  • número_cpf
  • tem_convenio
  • telefone_contato
  • data_cadastro
Pedido · Delivery
  • numero_pedido
  • cliente_nome
  • endereco_entrega
  • valor_total
  • status_pedido
  • data_criacao
Veículo · Frota
  • placa
  • modelo
  • kilometragem
  • status_manutencao
  • motorista_responsavel
Importante: Cada atributo deve ter uma razão de existir no contexto do cliente real. Não inventem atributos "só porque sim".
Parte 2 · 2.2

Comportamentos: o que o Objeto "FAZ"

Comportamentos (ou métodos) são as ações que um objeto pode realizar. São funções que manipulam os dados (atributos) do objeto.

Paciente · Hospital
  • marcar_consulta
  • cancelar_consulta
  • atualizar_telefone
  • renovar_convenio
Pedido · Delivery
  • calcular_tempo_entrega
  • atualizar_status
  • cancelar_pedido
  • calcular_taxa_entrega
Veículo · Frota
  • registrar_manutencao
  • atualizar_kilometragem
  • liberar_para_viagem
  • retornar_garagem
Nota: Hoje, vamos modelar esses comportamentos em linguagem natural (português). Na Aula 4 e 5, vamos transformá-los em código Python real.

Parte 3

Modelagem Prática

Como identificar a entidade principal do seu sistema e definir seus atributos e comportamentos com clareza.

Parte 3 · 3.1

Escolhendo a Entidade Principal

A entidade principal é o "personagem" mais importante do negócio do cliente. Tudo no sistema gira em torno dela.

🏥 Hospital → Paciente
🍔 Delivery → Pedido
🚌 Frota → Veículo / Trajeto
🏫 Escola → Aluno / Turma
Como descobrir qual é a entidade principal?
  • "Qual é o 'objeto' que o cliente mais quer controlar e acompanhar?"
  • "Se o sistema desaparecesse, qual informação seria a mais valiosa para manter?"
  • "Em que tabela do banco de dados essa entidade se transformaria?"
Parte 3 · 3.2

Identificando Atributos Essenciais

Para a entidade principal do seu sistema, liste os atributos que realmente importam para o cliente.

Roteiro
  • Pergunte-se: "O que o cliente precisa GUARDAR sobre cada [sua entidade]?"
  • Liste entre 5 e 7 atributos essenciais.
  • Para cada atributo, pense: "Por quê? Para quê? Como o cliente usará essa informação?"

Exemplo – Paciente no Hospital

AtributoPor quê?
nomeIdentificar o paciente
cpfRegistro oficial único
idadeDeterminar riscos e tratamentos
tem_convenioDefinir tipo de atendimento e cobrança
telefoneEntrar em contato para consultas
data_cadastroControlar quanto tempo é paciente ativo
Parte 3 · 3.3

Identificando Comportamentos Essenciais

Para a mesma entidade, liste os comportamentos (ações) que ela precisa realizar.

Roteiro
  • Pense: "O que o cliente quer que o sistema FAÇA com cada [sua entidade]?"
  • Liste 5 a 7 comportamentos principais.
  • Use a estrutura: verbo + complemento — ex: marcar_consulta, atualizar_telefone

Exemplo – Paciente no Hospital

ComportamentoO que faz?
marcar_consultaAgenda uma consulta para o paciente em uma data específica
cancelar_consultaRemove a consulta da agenda
atualizar_telefoneAtualiza o número de contato para emergências
renovar_convenioRenova a validade do convênio de saúde

Parte 9

Dúvidas Frequentes

P: Posso ter mais de uma entidade principal?
Não, cada grupo deve escolher UMA entidade principal para começar. As outras virão depois (Pedido pode ter Cliente, Veículo pode ter Motorista). Mas a entidade principal é aquela que tudo gira em torno.
P: Quantos atributos a classe deve ter?
Entre 4 e 7 atributos essenciais. Se tiver menos, está incompleto. Se tiver mais de 10, você pode estar misturando conceitos.
P: Preciso escrever o código da docstring exatamente como nos exemplos?
Não! Os exemplos são modelos. Adapte para seu contexto e cliente real. Seja criativo e sincero sobre o que a classe faz.
P: E se eu não souber que comportamentos a classe deve ter?
Pense no fluxo real. Se é um paciente, ele pode marcar e cancelar consultas. Se é um pedido, pode ser cancelado ou rastreado. Comece simples; você pode adicionar mais na Aula 5.
P: O arquivo modelos.py fica para sempre ou é descartado depois?
Fica para sempre! Você vai expandir esse arquivo nas próximas aulas. Hoje é só o começo.
P: O que fazer se o Git estiver dando erro?
Chame o docente. Pode ser configuração ou repositório não inicializado. Melhor resolver agora do que arrastar para as próximas aulas.

Arquitetura em Camadas e o Padrão MVC

Modularização e Divisão de Responsabilidades.

Arquitetura em Camadas e o Padrão MVC

A. Conceito:

A Modularização e a Arquitetura em Camadas consistem na divisão de um sistema de software em partes menores e gerenciáveis, onde cada módulo assume uma responsabilidade única.

Isso se baseia no princípio SOLID (Single Responsibility Principle), que defende que um componente deve ter apenas um motivo para mudar.

Arquitetura em Camadas e o Padrão MVC

A. Conceito: (Continuação)

O MVC (Model-View-Controller) é um padrão arquitetural que separa o sistema em três grandes frentes: a apresentação (View), o controle do fluxo da aplicação (Controller) e a manipulação das regras de negócio e dados (Model).

Nesse modelo, o acoplamento é reduzido e a coesão é aumentada. Nós não misturamos o código que desenha um botão na tela com a sintaxe que executa uma inserção no banco de dados.

Essa separação em camadas permite um trabalho em equipe mais eficiente, já que desenvolvedores podem atuar em frentes distintas (Frontend vs Backend) sem causar conflitos diretos no código fonte, refletindo uma organização de trabalho padronizada.

Arquitetura em Camadas e o Padrão MVC

B. Explicação:

Lembrando o que é Modularização?

  • Divisão de um sistema grande em partes menores e gerenciáveis
  • Cada módulo tem uma responsabilidade específica

Aqui está o "quem é quem" nessa estrutura MVC:

Estrutura MVC

Model (Modelo)

O "coração" da aplicação. É aqui que os dados residem e onde as regras de negócio são aplicadas.

  • Responsabilidade: Gerenciar o estado dos dados, fazer cálculos, validar informações e interagir com o banco de dados.
  • Exemplo: Em um app de banco, o Model sabe como calcular juros ou verificar se você tem saldo suficiente para um saque.
Estrutura MVC

View (Visão)

A interface que o usuário vê e com a qual interage.

  • Responsabilidade: Apresentar os dados vindos do Model de uma forma legível (HTML, telas de app, gráficos). Ela não deve conter lógica complexa, apenas o necessário para exibir a informação.
  • Exemplo: O formulário onde você digita seu login ou a lista de produtos em um site de compras.
Estrutura MVC

Controller (Controlador)

O "maestro" ou intermediário. Ele recebe as ações do usuário e decide o que deve ser feito.

  • Responsabilidade: Capturar o clique do botão ou o texto digitado na View, enviar para o Model processar e, depois, dizer à View qual resultado mostrar.
  • Exemplo: Quando você clica em "Salvar", o Controller pega os dados do formulário, envia para o Model e redireciona você para uma página de "Sucesso".
Arquitetura em Camadas e o Padrão MVC

O Fluxo Resumido

  1. O Usuário interage com a View (ex: clica em um botão).
  2. O View notifica o Controller que recebe essa ação.
  3. O Controller solicita a ação ao Model.
  4. O Model processa, altera seu estado e notifica o Controller.
  5. O Controller envia a notificação atualizada para o View.
  6. A View é atualizada para exibir os novos dados ao usuário.
Arquitetura em Camadas e o Padrão MVC

Por que usar o MVC?

A grande vantagem é a manutenibilidade. Imagine que você queira mudar o visual do seu site (a View) para uma versão mobile. Se o seu código estiver em MVC, você pode trocar toda a interface sem precisar mexer em uma única linha da lógica de cálculo ou do banco de dados (o Model).

  • Reutilização: Você pode usar o mesmo Model para múltiplas View (um site ou um app mobile).
  • Testabilidade: Mais fácil testar componentes isoladamente.
  • Trabalho em equipe: Um designer pode focar na View enquanto um desenvolvedor foca no Model, sem que um "atropele" o código do outro.

O Padrão DAO (Data Access Object)

Persistência de Dados e Encapsulamento.

O Padrão DAO

A. Conceito:

O padrão DAO (Data Access Object) é a materialização extrema do encapsulamento no nível de persistência de dados. Ele age como um componente dentro da camada Model, isolando completamente o Banco de Dados do restante da aplicação.

Toda a complexidade do SQL e o uso da biblioteca sqlite3 ficam restritos dentro das classes DAO (ex: ProdutoDAO, ClienteDAO).

O Padrão DAO

A. Conceito: (Continuação)

Com essa abordagem técnica, o nosso main.py passa apenas a invocar métodos simples como salvar() ou buscar_todos(), lidando somente com Objetos Python, não com tabelas relacionais.

A grande vantagem estratégica é o baixo acoplamento estrutural: podemos refatorar consultas pesadas, otimizar índices no banco ou trocar o SGBD para outro modificando exclusivamente o arquivo DAO, mitigando riscos e promovendo entregas de software altamente íntegras.

Evolução da Arquitetura

B. Evolução da Arquitetura e Limitações do MVC Simples:

O padrão MVC básico (Model-View-Controller) funciona bem em aplicações pequenas, mas apresenta desafios importantes conforme o projeto cresce:

  • Concentração de lógica no Controller: Quando regras de negócio são complexas, o Controller fica inchado e responsável por múltiplas funções
  • Impossibilidade de reutilização: Uma regra de negócio implementada em um Controller não pode ser facilmente utilizada em outro Controller
  • Acoplamento com banco de dados: A lógica de negócio fica misturada com comandos SQL, dificultando testes e manutenção
  • Dificuldade na migração: Trocar de MySQL para PostgreSQL ou outro banco requer alterações em múltiplos pontos da aplicação
Evolução da Arquitetura

Solução: Arquitetura em Camadas.

A resposta para essas limitações é a separação em mais camadas, cada uma com responsabilidade bem definida e clara.

O DAO esconde a "sujeira" do SQL e das conexões, permitindo que o resto do seu programa fale apenas a linguagem de objetos.

Conceitos Essenciais

D. E o tal construtor __init__?

O construtor __init__ no Python é como a "linha de montagem inicial" de um objeto. Ele serve para inicializar o estado de um objeto no exato momento em que ele é criado (instanciado).

Lembra que a classe é como uma fôrma de bolo (o molde). O __init__ é o momento em que você despeja a massa na fôrma e define qual será o sabor daquele bolo específico antes de colocá-lo no forno.

Conceitos Essenciais

Como funciona na prática?

Sempre que você "chama" uma classe para criar um novo objeto, o Python executa o método __init__ automaticamente, por debaixo dos panos. Ele garante que o objeto já nasça com todas as características (atributos) essenciais para funcionar.

Vamos olhar para o exemplo da classe Cliente:

Python class Cliente: def __init__(self, id, nome, email): self.id = id self.nome = nome self.email = email
Conceitos Essenciais

Instanciando o Objeto

Quando você digita isso no seu código:

Python cliente_ana = Cliente(1, "Ana", "ana@senai.br")

Aqui está o passo a passo do que o init faz nos bastidores:

  1. O Python cria um espaço vazio na memória para o novo cliente.
  2. Ele chama o método __init__ e passa esse espaço vazio através da palavra self (que significa "eu mesmo" ou "este próprio objeto").
  3. O __init__ pega os valores que você forneceu (1, "Ana", "ana@senai.br") e os guarda dentro do objeto (self.id = id, etc.).
  4. O objeto nasce pronto e preenchido, e é entregue para a variável cliente_ana. {ver img03.png}
Conceitos Essenciais

Por que ele é tão importante?

  • Garantia de consistência: Ele impede que você crie um "Cliente pela metade". Se a sua regra diz que todo cliente precisa de um ID, Nome e Email, o __init__ obriga o programador a fornecer esses dados na hora da criação. O código vai dar erro se você tentar criar apenas com Cliente(1, "Ana").
  • Organização: Evita que você crie um objeto vazio para depois ter que preencher atributo por atributo em várias linhas de código diferentes.
  • Configuração inicial: Além de receber dados externos, o __init__ é ótimo para configurar valores padrão. Por exemplo, você poderia ter um self.ativo = True dentro do __init__, fazendo com que todo novo cliente já nasça ativado no sistema, sem precisar passar isso no momento da criação.
O Padrão DAO

E. Exemplo de Código:

Python import sqlite3 from models.produto import Produto class ProdutoDAO: """Encapsula toda a interação com o banco de dados para a entidade Produto.""" def __init__(self, db_path: str = "banco_sa.db"): self.db_path = db_path def salvar(self, produto: Produto): """Recebe um objeto e faz a persistência via SQL.""" conexao = sqlite3.connect(self.db_path) cursor = conexao.cursor() sql = "INSERT INTO produtos (id, nome, preco) VALUES (?, ?, ?)" valores = (produto.id_produto, produto.nome, produto.preco) try: cursor.execute(sql, valores) conexao.commit() print("LOG: Produto salvo com sucesso no banco de dados.") except sqlite3.Error as e: print(f"ERRO DAO: Falha ao inserir produto. Detalhes: {e}") finally: conexao.close()
O Padrão DAO

F. Explicação linha por linha:

  • Linhas 1-2: Importamos o módulo sqlite3 e a nossa classe de modelo Produto. O DAO precisa conhecer a entidade que ele vai gerenciar.
  • Linhas 4-8: Criamos a classe ProdutoDAO. Seu construtor define qual banco de dados acessar, padronizando a conexão.
  • Linha 10: O método salvar não recebe variáveis soltas (nome, preco), ele recebe a Entidade completa produto: Produto. Isso é Programação Orientada a Objetos aplicada.
  • Linhas 12-13: Abrimos a conexão com o banco e preparamos o cursor, que será nosso "carteiro" executando as queries.
O Padrão DAO

F. Explicação linha por linha (Continuação):

  • Linhas 15-16: Preparamos a nossa string SQL limpa. Usamos ? como bind parameters para blindar nosso sistema contra ataques de SQL Injection, demonstrando segurança. A tupla valores extrai os dados direto do objeto recebido.
  • Linhas 18-25: Bloco try/except/finally. Tentamos executar e confirmar a transação (commit). Se o SGBD retornar erro (ex: chave primária duplicada), capturamos a exceção educadamente sem travar o sistema. O bloco finally garante a boa prática de fechar a conexão sempre, economizando memória.

FAQ - Dúvidas Reais

Perguntas frequentes sobre o conteúdo da aula.

FAQ

Dúvidas Reais

1. Por que criar tantos arquivos e pastas (models, views, dao)? Não seria mais rápido colocar tudo num arquivo só?
FAQ

Dúvidas Reais

Resposta: Para um trabalho acadêmico pequeno, sim. Mas na indústria real, projetos escalam. Com milhares de linhas, rastrear um erro num único arquivo toma horas. A separação garante organização mental para a equipe e facilita o controle de versão (Git) sem gerar conflitos simultâneos.
FAQ

Dúvidas Reais

2. Posso usar o input() dentro do DAO para perguntar ao usuário qual dado ele quer deletar?
FAQ

Dúvidas Reais

Resposta: De jeito nenhum! O DAO é estritamente técnico e não tem interface com o usuário. A View deve perguntar o ID, repassar para o Controller, que então chama dao.deletar(id_escolhido). Misturar input() no DAO destrói o conceito da arquitetura.
FAQ

Dúvidas Reais

3. Se eu precisar usar MySQL em vez de SQLite, o que acontece com os objetos Modelo?
FAQ

Dúvidas Reais

Resposta: Nada! Eles continuam intactos. Esse é o poder do encapsulamento. A única alteração necessária será dentro da classe DAO para usar o driver do MySQL (ex: mysql.connector), mantendo o resto do ecossistema a salvo.
FAQ

Dúvidas Reais

4. Para toda tabela no banco de dados eu terei que criar uma classe Modelo e uma classe DAO?
FAQ

Dúvidas Reais

Resposta: Exatamente. Esse é o padrão de mercado. Se você tem tabelas para Clientes, Fornecedores e Vendas, você terá ClienteDAO, FornecedorDAO e VendaDAO. Isso divide a complexidade do seu SGBD de forma elegante em código.
FAQ

Dúvidas Reais

5. No método try/except, o que acontece se ocorrer um erro e a conexão ficar aberta no banco?
FAQ

Dúvidas Reais

Resposta: Isso geraria um vazamento de recursos (memory leak) e eventualmente o banco "travaria" (database locked). É por isso que sempre colocamos conexao.close() dentro do bloco finally, que é executado ocorrendo erro ou não, garantindo integridade e estabilidade.

01
Parte Um

O que é um SGBD, "de verdade"?

Fundamentos

O Cérebro Organizado

Pense em qualquer sistema que você usa e que "lembra" de você: aplicativo de banco, Netflix, Instagram, sistema da escola, sistema de mercado, etc. Todos eles precisam guardar informação em algum lugar de forma organizada.

Esse "cérebro organizado" é o banco de dados; o SGBD é o programa especializado que cuida desse cérebro.

Analogia

1.1 Metáfora do Arquivista

Analogia: Imagine um armário grande cheio de pastas de alunos: cada pasta tem boletins, faltas, documentos, fotos.
• Banco de dados = o armário + as pastas.
• SGBD = o arquivista profissional da escola.

O professor não entra na sala dos arquivos para mexer direto nas pastas. Você pede para o arquivista: "Me traga o histórico escolar do aluno João, matrícula 123."

Processamento

O que o Arquivista verifica?

  • Se você tem permissão para ver aquele arquivo.
  • Onde está a pasta do João.
  • Se não tem outra pessoa mexendo na mesma pasta ao mesmo tempo.
  • Depois devolve o documento certo, na hora certa.
Fluxo de Dados

Na informática, o SGBD faz isso:

1
Pedido
Recebe um pedido da aplicação (ex: "buscar aluno com matrícula 123").
2
Linguagem
Entende a linguagem desse pedido (SQL, por exemplo).
3
Hardware
Vai nos "bits e bytes" do disco rígido buscar a informação e entrega de volta formatada.
Arquitetura

1.2 Ponte entre Software e Hardware

O SGBD fica no meio do caminho. De um lado, você tem os arquivos binários gravados no disco (difíceis de ler manualmente). Do outro lado, você tem o programador e o usuário final.

Sem o SGBD:

O programador teria que escrever código para abrir o arquivo em baixo nível, procurar a linha certa e garantir que o arquivo não corrompesse.

Parte Dois

Funções fundamentais de um SGBD

Segurança

2.1 Controle de acesso

O SGBD garante que apenas usuários autorizados vejam ou alterem dados.

  • Exemplo: Em um sistema escolar, o aluno pode ver sua nota (SELECT), mas não pode alterá-la (UPDATE). Somente o professor ou a secretaria têm essa permissão.
Flexibilidade

2.2 Independência de dados

Isso significa que a forma como o dado é guardado no disco (física) é separada de como o programador vê o dado (lógica).

Se o administrador do banco decidir trocar o HD por um SSD ou mudar o formato do arquivo para ganhar velocidade, o programador não precisa mudar nem uma linha do seu código SQL. O SGBD cuida da "tradução".

Otimização

2.3 Redundância controlada

Em arquivos comuns, é fácil repetir a mesma informação várias vezes (ex.: o nome do cliente em cada arquivo de pedido).

Nome do Cliente em toda planilha de Pedido
ID do Cliente relacionando tabelas

O SGBD ajuda a organizar os dados para que cada pedaço de informação seja guardado em apenas um lugar, evitando que, ao atualizar o endereço de um cliente, você esqueça de atualizar em algum arquivo antigo.

Qualidade

2.4 Integridade de dados

O SGBD impõe regras para garantir que os dados façam sentido.

  • Impedir que uma nota seja maior que 10.
  • Impedir que existam dois alunos com o mesmo CPF.
  • Impedir que um pedido seja feito para um cliente que não existe.
Confiança

2.5 Persistência e Concorrência

Persistência: Garante que, uma vez que o dado foi salvo, ele não se perca mesmo que o computador desligue sozinho por falta de energia.

Controle de Concorrência: Se duas pessoas tentam comprar o último ingresso de cinema ao mesmo tempo, o SGBD garante que apenas uma consiga, criando uma "fila" invisível para evitar erros.

Parte Três

O que são as propriedades ACID?

ACID: A

3.1 Atomicidade (Tudo ou Nada)

Uma transação é uma sequência de operações. A atomicidade garante que ou todas funcionam, ou nenhuma é gravada.

Exemplo bancário: Se você transfere dinheiro, o banco precisa 1) tirar da sua conta e 2) colocar na outra. Se a luz cair entre o passo 1 e o 2, o SGBD cancela tudo para você não perder dinheiro.
ACID: C

3.2 Consistência

O banco garante que as regras definidas (integridade) nunca sejam quebradas. Se você tentou inserir um dado inválido, o banco rejeita a transação inteira para manter o sistema "saudável".

ACID: I

3.3 Isolamento

Enquanto uma pessoa está atualizando um dado, outra pessoa que tentar ler esse mesmo dado verá a versão anterior até que a atualização termine. Isso evita que alguém veja um dado "pela metade".

ACID: D

3.4 Durabilidade

Uma vez que o SGBD confirmou que a operação foi realizada ("Commit"), o dado está gravado permanentemente. Mesmo que o servidor trave logo em seguida, o dado estará lá quando ele voltar.

Parte Quatro

Arquitetura Relacional (SQL)

Estrutura SQL

4.1 Organização em Tabelas

A ideia central é que tudo é uma tabela (formalmente chamada de Relação).

  • Linhas (Tuplas): Representam um registro único (um aluno, um produto).
  • Colunas (Atributos): Representam as características (nome, preço, CPF).

É muito parecido com o Excel, mas com regras matemáticas rigorosas por trás.

Modelagem

4.2 Schema Rígido

No modelo relacional, você precisa definir a estrutura antes de colocar os dados.

Se você definiu que a tabela "Alunos" tem Nome, Email e CPF, você não pode simplesmente tentar salvar o Instagram do aluno se essa coluna não tiver sido criada antes.

Relacionamentos

4.3 Chaves Primárias e Estrangeiras

  • Chave Primária (PK): O documento de identidade da linha (ex.: ID do aluno). Não pode repetir.
  • Chave Estrangeira (FK): É o que liga uma tabela à outra.
Exemplo: Na tabela de "Notas", você não escreve o nome do aluno, você coloca apenas o ID dele (FK). Assim, o SGBD sabe exatamente a quem aquela nota pertence.

Parte Cinco

Arquitetura Não Relacional (NoSQL)

Evolução

5.1 Por que surgiu?

Com o crescimento gigante da web (Google, Facebook, Amazon), as tabelas rígidas do SQL começaram a ter dificuldades para escalar e lidar com dados que mudam o tempo todo.

O NoSQL (Not Only SQL) foca em flexibilidade e velocidade para volumes massivos de dados.

Categorias

6. Tipos comuns de NoSQL

Diferente do SQL que só tem tabelas, o NoSQL tem vários formatos:

  • Documentos (Ex: MongoDB): Salva os dados como se fossem "pastas digitais" (JSON). Útil para perfis de usuários onde cada um pode ter informações diferentes.
  • Chave-Valor (Ex: Redis): Como um dicionário. Você dá uma chave e ele traz o valor instantaneamente. Muito usado para "carrinhos de compras" ou sessões de login.
  • Grafos (Ex: Neo4j): Focado em conexões. Ideal para redes sociais ou sistemas de recomendação.
Comparativo

Prática

7.2 Pensando no projeto da turma

Perguntas que ajudam a escolher:

?
Os dados têm estrutura estável? (Ex.: alunos, notas). SQL faz sentido.
?
Precisa escalar para milhões e dados mudam sempre? (Ex.: logs, rede social). NoSQL pode ser vantajoso.
?
Integridade e consistência rígida são críticas? (Ex.: financeiro). SQL é o padrão.

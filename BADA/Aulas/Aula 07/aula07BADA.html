Introdução

O que é o Modelo Lógico (DER) e por que ele é crucial.

Seção 1 · Introdução

1. Introdução

Cenário Real
Pensem em gigantes como Amazon ou iFood. Na aula passada, vimos que essas empresas dependem de uma modelagem de dados impecável, pois um erro no diagrama hoje significa um sistema lento ou dados duplicados amanhã.

Nós já desenhamos o nosso "mapa" abstrato, o Modelo Entidade-Relacionamento (MER), que documenta as regras de negócio.

Mas um banco de dados real não lê desenhos ou conceitos abstratos; ele lê tabelas concretas.

O passo de hoje é exatamente a ponte que transforma a nossa ideia de negócio em uma estrutura que a máquina compreende: o Modelo Lógico (DER).

Seção 1 · Problema Central

O Desafio dos Dados

Pergunta
Se o nosso banco de dados armazenar clientes e pedidos, como ele sabe exatamente qual cliente fez qual pedido em um universo de milhões de registros, sem misturar os dados de todo mundo?
Ação
Nós vamos analisar rapidamente uma planilha "bagunçada", onde o nome do cliente e os detalhes do pedido estão repetidos na mesma linha várias vezes.

Desenvolvimento e Prática

Transformação para Tabelas

Seção 2 · Desenvolvimento

Transformação para Tabelas - A. Conceito

Quando terminamos o Modelo Conceitual (MER), temos entidades (retângulos) e atributos (elipses).

O processo de transformação para o Modelo Lógico significa que cada entidade forte que desenhamos vai se tornar uma Tabela no nosso banco de dados.

Os atributos descritivos que pertencem a essa entidade se transformam nas Colunas dessa tabela.

Essa é a fundação para estruturar a informação de forma que um Sistema de Gerenciamento de Banco de Dados (SGBD) consiga gravar e recuperar os dados.

Seção 2 · Desenvolvimento

Transformação para Tabelas - Cuidados

Esse mapeamento precisa ser feito com cuidado. Um atributo que guarda o nome do cliente será uma coluna de texto (VARCHAR), enquanto a data de nascimento será uma coluna de data (DATE).

Transformar entidades em tabelas é o ato de dar um formato físico às nossas ideias de negócio, preparando o terreno para que o sistema de fato ganhe vida.

Seção 2 · Desenvolvimento

B. Explicação técnica

A tradução de conjuntos de entidades do modelo ER para o relacional resulta na criação de relações (tabelas) que contêm o mesmo número de colunas que os atributos simples da entidade original.

Os atributos compostos ou multivalorados exigem tratamentos específicos, mas a regra geral é o mapeamento direto de Entidade para Tabela.

Seção 2 · Desenvolvimento

O Roteiro DDL

O modelo lógico resultante deve ser livre de ambiguidades estruturais.

Ele não só define os nomes das tabelas e colunas, mas também os tipos de dados básicos, servindo de roteiro exato para os comandos de criação (DDL) que executaremos no SGBD.

Vamos traduzir esse conceito didaticamente em três partes principais:

Seção 2 · Desenvolvimento

1. A Regra de Ouro: Entidade vira Tabela

No MER (o desenho), nós temos Entidades (como "Cliente" ou "Produto") e Atributos (como "Nome" e "Preço"). A regra geral e mais simples da tradução é:

  • A Entidade inteira se transforma em uma Tabela.
  • Cada Atributo Simples se transforma em uma Coluna dessa tabela.

Se você tinha uma entidade chamada Cliente com os atributos simples CPF e Nome, você terá exatamente uma tabela Cliente com as colunas CPF e Nome.

Seção 2 · Desenvolvimento

2. Os Rebeldes: Atributos Compostos e Multivalorados

Alguns atributos exigem "tratamentos específicos". Isso acontece porque bancos de dados relacionais odeiam bagunça e exigem que cada "célula" da tabela tenha apenas um único valor.

  • Atributos Compostos: São aqueles que podem ser divididos. Por exemplo, "Endereço". O banco de dados não gosta de um campo gigante com "Rua X, Número 10, Bairro Y". A solução é quebrá-lo em colunas simples: Rua, Numero, Bairro.
Seção 2 · Desenvolvimento

Atributos Multivalorados

  • Atributos Multivalorados: São aqueles onde você pode ter vários valores ao mesmo tempo, como "Telefone" (uma pessoa pode ter celular e fixo). Não podemos colocar dois números na mesma célula. A solução geralmente é criar uma nova tabela só para os telefones, ligada ao cliente (usando aquele "fio invisível" que vimos antes!).
Seção 2 · Desenvolvimento

3. A Planta Baixa Exata (O Modelo Lógico e o DDL)

O modelo lógico deve ser "livre de ambiguidades estruturais" e serve de "roteiro exato" para o DDL.

O que isso significa? Significa que acabou a fase de rascunho.

No modelo lógico, não basta dizer "aqui vai o telefone".

Você precisa ser absurdamente específico, definindo os tipos de dados básicos:

Seção 2 · Desenvolvimento

Tipos de Dados

Nome VARCHAR(50) Aceita até 50 letras.
Idade INT Só aceita números inteiros.
Data_Nascimento DATE Tem que seguir o formato de calendário.
Seção 2 · Desenvolvimento

Conexão com DDL

Quando você deixa tudo assim, tão bem definido e sem ambiguidades, esse modelo serve como um roteiro direto para o DDL (Data Definition Language).

O DDL é a linguagem de comandos que о SGBD (Sistema de Gerenciamento de Banco de Dados, como MySQL ou PostgreSQL) usa para efetivamente criar a tabela.

É o famoso comando CREATE TABLE que os programadores digitam.

Chaves Primárias e Estrangeiras

Estabelecendo Identificadores e Relacionamentos.

Seção 3 · Chaves

A. Conceito de Chave Primária (PK)

Na última aula, vimos que o identificador é um atributo capaz de individualizar uma ocorrência dentro do sistema, seguindo regras de ouro: deve ser único e nunca pode ser nulo.

No Modelo Lógico, esse identificador recebe o nome técnico de Chave Primária (Primary Key - PK).

Ela é o "CPF" daquela linha dentro da tabela.

Seção 3 · Chaves

Chave Estrangeira (FK)

Para que as tabelas conversem entre si, usamos a Chave Estrangeira (Foreign Key - FK).

A FK é simplesmente a cópia da Chave Primária de uma tabela colocada dentro de outra tabela.

É assim que criamos os relacionamentos de 1:N (um para muitos).

Se um Cliente faz vários Pedidos, a tabela de Pedidos terá uma coluna (FK) guardando o ID do Cliente para sabermos a quem aquele pedido pertence.

Seção 3 · Chaves

B. Explicação técnica

A restrição de Chave Primária garante a integridade de entidade, assegurando que não haja registros duplicados.

A restrição de Chave Estrangeira garante a integridade referencial, o que impede que um registro aponte para uma chave inexistente (por exemplo, cadastrar um pedido para um cliente que não está no banco de dados).

Seção 3 · Chaves

Mapeamento 1:N

Quando mapeamos um relacionamento 1:N do modelo conceitual para o lógico, a regra de conversão dita que a chave primária do lado "1" migra como chave estrangeira para a tabela do lado "N".

Seção 3 · Chaves

Traduzindo as Regras

A Chave Primária (A Lei do CPF Único)
Traduzindo a "integridade de entidade". A Chave Primária (PK) é como o CPF daquela linha da tabela. Vocês já viram duas pessoas com o mesmo CPF? Não. A regra do banco de dados é a mesma: é terminantemente proibido ter dois clientes com o ID 'C-01'. Essa regra garante que ninguém seja confundido. Cada registro é único e exclusivo."
Seção 3 · Chaves

Traduzindo as Regras

A Chave Estrangeira (A Lei "Não Atendemos Fantasmas")
A "integridade referencial" conectando com a vida real. A Chave Estrangeira (FK) é aquele 'fio invisível que liga o Pedido ao Cliente. Mas o banco de dados tem um segurança na porta: ele não deixa você cadastrar uma venda para o cliente 'C-99' se esse cliente não existir na Tabela de Clientes. O sistema vai gritar: 'Erro! Você está tentando vender para um fantasma!'. Isso é a integridade referencial: você só pode apontar para algo que existe."
Seção 3 · Chaves

O Mapeamento 1:N (A Regra de "Quem carrega quem?")

Esta é a parte crucial. Por que a chave do lado 1 (Cliente) vai para a tabela do lado N (Pedidos) e não o contrário?

Use a lógica do espaço:

O Cenário: Temos uma relação 1:N. 1 Cliente pode fazer N (Muitos) Pedidos.

Pergunta: "Onde faz mais sentido guardar o fio invisível? Na tabela de Clientes ou na de Pedidos?"

Seção 3 · Chaves

Migração da Chave

  • Opção Errada (N migrando para o 1): "Se colocarmos a chave na tabela de Clientes, teríamos que criar infinitas colunas lá dentro: Pedido_1, Pedido_2, Pedido_30... A tabela ia estourar! O Cliente não consegue segurar todos os pedidos."
  • A Regra Certa (1 migrando para o N): "É mais fácil o Pedido carregar o 'crachá' do Cliente. Como cada pedido pertence a apenas um cliente, nós simplesmente colocamos o ID do Cliente dentro da tabela de Pedidos. É por isso que a chave primária do lado '1' sempre viaja e vira chave estrangeira no lado 'N'."
Seção 3 · Chaves

Resumindo

  • Chave Primária: "Eu sou único."
  • Chave Estrangeira: "Eu pertenço a alguém que existe de verdade."
  • Regra 1:N: "O filho (N) carrega o sobrenome do pai (1)."

Seção 3 · Chaves

D. Exemplo de Código

SQL Criando a tabela de Pedidos com Chave Estrangeira CREATE TABLE Pedido ( id_pedido INT PRIMARY KEY AUTO_INCREMENT, data_pedido DATE NOT NULL, valor DECIMAL(10,2) NOT NULL, fk_cliente VARCHAR(11), FOREIGN KEY (fk_cliente) REFERENCES Cliente(cpf) );
Seção 3 · Chaves

E. Explicação linha por linha:

  • id_pedido INT PRIMARY KEY AUTO_INCREMENT,: Cria a Chave Primária da tabela pedido. O banco vai gerar esse número automaticamente.
  • data_pedido... e valor...: Colunas comuns do pedido.
  • fk_cliente VARCHAR(11),: Cria a coluna que vai receber a informação do cliente (ela deve ter o mesmo tipo de dado da PK do cliente).
  • FOREIGN KEY (fk_cliente) REFERENCES Cliente (cpf): A mágica acontece aqui! O banco cria a restrição (o fio invisível) dizendo que a coluna fk_cliente deve obrigatoriamente referenciar um cpf válido lá na tabela Cliente.

Relacionamentos N:N

Resolvendo o Muitos para Muitos

Seção 4 · N:N

A. Conceito

Às vezes, as regras de negócio criam teias complexas. Um Aluno cursa várias Disciplinas, e uma Disciplina tem vários Alunos.

Isso é um relacionamento Muitos para Muitos (N:N). O grande problema é que bancos de dados relacionais não aceitam relacionamentos N:N diretos nas tabelas, pois isso violaria as regras de normalização (não podemos guardar uma lista de valores dentro de uma única célula da tabela).

Seção 4 · N:N

A Solução Associativa

Para resolver isso de forma inteligente, nós criamos uma "Tabela Associativa" no meio das duas.

Essa nova tabela herda as chaves primárias de ambas as tabelas originais, transformando-as em chaves estrangeiras.

O que antes era um problema intransponível se quebra em dois relacionamentos de 1:N muito fáceis de gerenciar!

Seção 4 · N:N

B. Explicação técnica

A conversão de conjuntos de relacionamentos M:N exige a criação de uma nova relação (tabela) para representar a associação.

A chave primária desta nova tabela é tipicamente composta pela combinação das chaves estrangeiras que referenciam as tabelas originais participantes.

Qualquer atributo que estava alocado no losango de relacionamento do MER (por exemplo, a "nota" de um aluno em uma disciplina) migra e se torna uma coluna nesta nova tabela associativa.

Seção 4 · N:N

Analogia Prática

Esse é o famoso cenário do "Muitos para Muitos" (M:N), que costuma ser o último grande obstáculo no aprendizado de modelagem.

Para explicar isso de forma leve e sem dar dor de cabeça na turma, vamos usar uma analogia que eles vivem na pele todos os dias: Alunos e Disciplinas.

Aqui está uma forma estruturada e didática de apresentar esse conceito no quadro ou nos slides:

Seção 4 · N:N

O Impasse do "Muitos para Muitos" (O Problema)

Temos a Tabela Aluno e a Tabela Disciplina.

  • 1 Aluno cursa várias Disciplinas.
  • 1 Disciplina tem vários Alunos.
Pergunta
"Onde nós vamos plugar o nosso 'fio invisível' (a Chave Estrangeira)? Se colocarmos dentro do Aluno, ele só vai poder se matricular em uma única matéria. Se colocarmos dentro da Disciplina, a matéria só vai poder ter um único aluno matriculado. E agora? Deu empate!"
Seção 4 · N:N

A Tabela "Casamenteira" (A Solução)

Bancos de dados não lidam bem com empates. Quando temos um relacionamento M:N, precisamos chamar um "mediador".

A regra de ouro aqui é: Muitos para Muitos sempre gera uma tabela nova.

Nós criamos uma terceira tabela, que podemos chamar de Tabela Associativa (ou 'Tabela Casamenteira').

O único trabalho dela na vida é cruzar os fios invisíveis. Vamos chamá-la de Matricula.

Seção 4 · N:N

O Atributo "Órfão" (Onde entra a Nota?)

E a tal da 'Nota'? Pense bem: a nota 8,0 pertence só ao João? Não, em Programação ele tirou 5,0.

A nota pertence só à disciplina de Banco de Dados? Não, a Maria tirou 10,0 lá.

A nota não é do aluno, nem da matéria. A nota é do evento, do momento em que o João se encontrou com o Banco de Dados. Por isso, tudo o que estava no losango do desenho (o relacionamento em si) ganha uma coluna nova e vai morar nessa nossa tabela associativa."

Seção 4 · N:N

A Chave Composta (O Super-Combo)

"A chave primária é tipicamente composta pela combinação das chaves"

Para fechar, lembram da regra de que toda tabela precisa de uma Chave Primária (PK) única, como um CPF?

Nossa tabela 'Matricula' não tem uma coluna de ID próprio. O que ela faz? Ela junta as duas chaves estrangeiras!

O banco de dados checa o 'combo': A-01+D-01. Essa combinação vira a nossa Chave Primária Composta. Isso é genial porque impede que o sistema cometa um erro grotesco: matricular o João duas vezes na exata mesma matéria no mesmo semestre.

C. Pensamento Visual

Pensem em duas ilhas separadas: Ilha ALUNO e Ilha DISCIPLINA.

O relacionamento N:N é como tentar pular de uma para a outra sem sucesso.

A solução é construir uma ponte (Tabela Associativa) chamada MATRICULA no meio do rio.

Agora, a ilha Aluno liga para a ponte (1:N), e a ilha Disciplina liga para a ponte (1:N).

D. Exemplo de Código

SQL Criando a tabela associativa (Ponte) CREATE TABLE Aluno_Disciplina ( fk_aluno INT, fk_disciplina INT, nota_final DECIMAL(4,2), PRIMARY KEY (fk_aluno, fk_disciplina), FOREIGN KEY (fk_aluno) REFERENCES Aluno(id_aluno), FOREIGN KEY (fk_disciplina) REFERENCES Disciplina(id_disciplina) );
Seção 4 · N:N

E. Explicação linha por linha:

  • fk_aluno INT, e fk_disciplina INT,: Criamos as colunas que receberão os IDs originais.
  • nota_final...: Atributo próprio do relacionamento.
  • PRIMARY KEY (fk_aluno, fk_disciplina),: Chave Primária Composta! A combinação do aluno com a disciplina é única, garantindo que o mesmo aluno não se matricule duas vezes na mesma matéria.
  • FOREIGN KEY... REFERENCES...: Cria as duas restrições garantindo que o aluno exista e a disciplina também.

FAQ - Dúvidas Reais

Perguntas Frequentes sobre Modelagem.

Seção 5 · FAQ

FAQ 01

1. Qual a diferença prática entre o MER e o DER?

O MER é conceitual e focado nas regras de negócio (desenho voltado para o cliente e analista). Ο DER (Modelo Lógico) é técnico, voltado para o banco de dados, mostrando PKs, FKs e os tipos de dados precisos.
Seção 5 · FAQ

FAQ 02

2. Posso ter mais de uma Chave Primária (PK) em uma tabela?

Não! Uma tabela só pode ter UMA chave primária. No entanto, como vimos no caso do N:N, essa única PK pode ser "composta" por duas ou mais colunas juntas.
Seção 5 · FAQ

FAQ 03

3. Toda tabela no meu sistema precisa ter uma Chave Estrangeira (FK)?

Não. Tabelas de domínio independentes (como Categorias, Cargos ou Cidades) muitas vezes só possuem a Chave Primária e fornecem dados para que outras tabelas usem sua FK.
Seção 5 · FAQ

FAQ 04

4. O que acontece se eu tentar apagar um registro de Cliente que já está atrelado a Pedidos (sendo FK)?

O SGBD vai bloquear a ação e lançar um erro de restrição de chave. Você precisará lidar com os pedidos primeiro (apagando-os ou inativando-os) antes de apagar o cliente, protegendo o banco contra "dados órfãos".
Seção 5 · FAQ

FAQ 05

5. Por que não posso deixar um relacionamento N:N direto no banco?

Porque bancos relacionais armazenam valores atômicos por coluna (apenas uma informação). Colocar múltiplos IDs em uma única célula impede o banco de fazer consultas eficientes e quebra as regras de Normalização.

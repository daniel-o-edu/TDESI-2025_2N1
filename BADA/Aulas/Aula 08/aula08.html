01

Dependência Funcional (DF)

O alicerce da integridade dos dados.

Dependência Funcional

A. Conceito

O que é Dependência Funcional (DF)?

De forma simples, a DF ocorre quando o valor de uma coluna define obrigatoriamente o valor de outra coluna. Se você conhece o "X", você automaticamente descobre o "Y".

A Dependência Funcional é a regra que diz que certas informações estão "amarradas" umas às outras. Entender essas amarras é o que permite organizar os dados em tabelas perfeitas (Normalização).

Dependência Funcional

1. A Estrutura (X -> Y)

  • X (Determinante): É a "chave". Ele manda na relação.
  • Y (Dependente): É a informação que "segue" o determinante.

2. Exemplo Prático: O CPF

Pense em uma tabela de pessoas: Se eu te der o número do CPF (X), você consegue me dizer exatamente qual é o Nome (Y) daquela pessoa.

Dizemos que: CPF → Nome.

O contrário nem sempre é verdade (existem muitas pessoas com o mesmo nome, então o Nome não determina o CPF).

Dependência Funcional

3. Por que isso é o "Alicerce"?

A DF serve para garantir a saúde dos dados. Se não respeitarmos essas regras, o banco de dados adoece com as chamadas anomalias:

  • Repetição desnecessária: Escrever a mesma informação várias vezes.
  • Erro de atualização: Mudar o nome de uma pessoa em uma linha, mas esquecer de mudar na outra (causando confusão).
  • Exclusão acidental: Apagar um dado e acabar perdendo outra informação importante por tabela.

A Normalização

A Regra de Ouro: Um lugar para cada coisa.

A Normalização

Conceito e Regra de Ouro

A Normalização é o processo de organizar as colunas e tabelas para garantir que as Dependências Funcionais sejam respeitadas. O objetivo é evitar a redundância (repetir dados) e as anomalias.

A Regra de Ouro: "Um lugar para cada coisa"

Em um banco de dados bem projetado, cada fato deve ser armazenado em apenas um lugar. Se um atributo depende de uma chave, ele deve estar na tabela daquela chave.

A Normalização

As 3 Primeiras Formas Normais

Para um esquema ser considerado "virtuoso", ele geralmente segue estes níveis:

    1ª Forma Normal (1FN): Sem grupos repetidos. Cada célula tem apenas um valor (nada de "Lista de telefones" em uma única célula).

    2ª Forma Normal (2FN): Já está na 1FN e todos os atributos dependem da chave inteira, não apenas de parte dela (importante em chaves compostas).

    3ª Forma Normal (3FN): Já está na 2FN e não existem dependências "transitivas" (tipo: A determina B, que determina C. O C deve sair dali e ir para outra tabela).

03

Primeira Forma Normal (1FN)

Atomicidade e a eliminação de grupos repetitivos.

1ª Forma Normal (1FN)

A. Conceito: Atomicidade

A 1FN exige uma única regra de ouro: a atomicidade. Todo cruzamento de linha e coluna deve conter um dado atômico, ou seja, único e indivisível, eliminando qualquer grupo repetitivo ou atributo multivalorado.

Não podemos ter uma célula com vários e-mails ou telefones. Quando detectamos um atributo multivalorado, como um Telefone, devemos transformar em uma nova entidade (ou tabela auxiliar).

O design deve seguir a regra do dado mínimo: "tudo o que é necessário está lá, e tudo o que está lá é necessário".

C. Exemplo de Código

-- TABELA FORA DA 1FN (ERRADO: Atributo multivalorado) CREATE TABLE Cliente (ID INT, Nome VARCHAR(50), Telefones VARCHAR(100)); -- TABELA NA 1FN (CORRETO: Decomposição sem perdas) CREATE TABLE Cliente ( ID_Cliente INT PRIMARY KEY, Nome VARCHAR(50) NOT NULL ); CREATE TABLE Cliente_Telefone ( ID_Telefone INT PRIMARY KEY AUTO_INCREMENT, ID_Cliente INT, Numero VARCHAR(15) NOT NULL, FOREIGN KEY (ID_Cliente) REFERENCES Cliente(ID_Cliente) );
1ª Forma Normal (1FN)

D. Explicação linha por linha

  • Linhas 5-8: Estruturamos a entidade Cliente apenas com atributos atômicos (indivisíveis). Retiramos os telefones daqui.

  • Linha 10: Criamos uma nova entidade relacional para tratar a multiplicidade de contatos.

  • Linhas 11-13: Definimos uma PK própria para o telefone (ID_Telefone) e guardamos o número de forma puramente atômica (Numero).

  • Linha 14: Aplicamos a Foreign Key para amarrar cada telefone registrado à sua entidade de origem, garantindo a decomposição sem perdas de informação.

04

Segunda Forma Normal (2FN)

O fim da Dependência Parcial.

2ª Forma Normal (2FN)

A. Conceito: Dependência Parcial

O teste e aprofundamento da 2FN só podem ser aplicados a tabelas que já estão na 1FN e que possuem obrigatoriamente Chaves Primárias Compostas (duas ou mais colunas formando a PK).

O objetivo da 2FN é erradicar a Dependência Parcial. Isso significa que todos os atributos não-chave da tabela devem depender da chave primária inteira, e não apenas de um "pedaço" dela.

Se um campo como "Nome do Produto" depende apenas do "ID_Produto" e ignora o "ID_Pedido" em uma tabela associativa, nós dividimos a tabela.

2ª Forma Normal (2FN)

B. Visualizando

Imaginem a entidade relacional Item_Venda com a chave composta (ID_Venda, ID_Produto). Se a coluna Descricao_Produto estiver dentro dessa tabela, ela depende parcialmente da chave (apenas do ID do Produto).

A normalização arranca a descrição daí e cria a tabela mãe Produto isolada.

2ª Forma Normal (2FN)

B. Visualizando

2ª Forma Normal (2FN)

C. Exemplo de Código

-- TABELA EM 1FN, MAS FORA DA 2FN (Dependência parcial existente) -- PK Composta: (ID_Venda, ID_Produto) -> 'Nome_Produto' depende só do Produto. -- REFATORANDO PARA 2FN: CREATE TABLE Produto ( ID_Produto INT PRIMARY KEY, Nome_Produto VARCHAR(100) NOT NULL -- Dependência TOTAL da PK ); CREATE TABLE Item_Venda ( ID_Venda INT, ID_Produto INT, Quantidade INT NOT NULL, -- Depende da chave composta INTEIRA! PRIMARY KEY (ID_Venda, ID_Produto), FOREIGN KEY (ID_Produto) REFERENCES Produto(ID_Produto) );
2ª Forma Normal (2FN)

D. Explicação linha por linha

  • Linhas 5-8: Isola-se o atributo Nome_Produto em sua respectiva entidade mestre, pois ele dependia apenas parcialmente da chave do pedido original.

  • Linha 10: Criamos a tabela onde ocorrerá a associação transacional.

  • Linhas 11-12: Definimos as chaves que irão compor a transação.

  • Linha 13: O atributo Quantidade permanece aqui porque ele só tem sentido e dependência funcional com a existência de ambas as chaves combinadas.

  • Linhas 14-15: Configuramos a PK Composta e asseguramos o relacionamento (FK).

FAQ - Dúvidas Reais

Esclarecendo pontos comuns sobre normalização.

FAQ
1. Podemos pular direto para a 2FN porque é "mais rápido"?
Resposta: Nunca. A normalização é um processo estrito e iterativo. A 2FN só é válida porque a tabela já teve seus valores convertidos em dados atômicos na 1FN.
FAQ
2. Como sei se preciso criar uma nova tabela para um atributo?
Resposta: Observe o domínio da informação. O cliente pode ter mais de um telefone? Se sim, é multivalorado. A 1FN exige a criação da entidade fraca/auxiliar conectada por FK.
FAQ
3. Muitas tabelas geradas na normalização não deixam a consulta (SELECT) mais lenta devido aos JOINS?
Resposta: É uma balança técnica. No projeto Lógico, nós evitamos anomalias e redundâncias através das FNs. Se houver lentidão física futura, aplicam-se "desnormalizações intencionais". Mas primeiro: normalize!
FAQ
4. Devo decorar essas formas matemáticas $(X \rightarrow Y)$?
Resposta: Não deve decorar, deve entender o princípio. O determinante é o identificador, e o dependente é a informação gerada a partir dele. Isso moldará seu raciocínio lógico em qualquer SGBD.
FAQ
5. A Inteligência Artificial pode modelar meu banco de dados sozinha?
Resposta: A IA generativa consegue ler estruturas e gerar excelentes DDLs. Mas quem possui o entendimento humano e lógico do que é uma "dependência parcial" na regra de negócios específica é o desenvolvedor. Use a IA para rascunhar, e o seu conhecimento para homologar e refinar.

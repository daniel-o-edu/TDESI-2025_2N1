Seção 2.1

Coleta e Sistematização de Dados — Teoria

A coleta de dados é o processo investigativo de extrair informações relevantes de um cenário de negócio ou dos requisitos de um cliente.

No contexto de desenvolvimento de sistemas, não podemos simplesmente começar a criar tabelas sem critério. Precisamos entrevistar os usuários (ou ler as especificações do projeto), analisar documentos existentes e observar como a informação flui.

A sistematização ocorre logo em seguida: é o ato de pegar esse amontoado de anotações, falas e regras de negócio e organizá-los de forma lógica, separando o que é processo do que é dado armazenável.

Por que isso importa
Essa etapa é crucial porque define o escopo do que o sistema irá gerenciar, garantindo que a aplicação funcional cumpra as expectativas do cliente de forma segura e eficiente.

Seção 2.2

Estruturação e Organização de Dados — Teoria

Após a coleta e a sistematização inicial, os dados precisam ser estruturados para a modelagem técnica. Estruturar dados significa padronizar a forma como eles serão tratados, garantindo que sejam compreensíveis para a equipe de desenvolvimento e, futuramente, para o Sistema de Gerenciamento de Banco de Dados (SGBD).

É neste momento que começamos a identificar padrões de repetição e dependências implícitas.

Exemplo prático
Em vez de registrar o endereço completo de um cliente em uma única linha de texto bagunçada, nós estruturamos isso em rua, número, bairro, cidade e estado. Essa organização de dados é a base das características de armazenamento que estudamos anteriormente.
Seção 2.2 · Demonstração Prática

Dados Não Estruturados vs. Estruturados

Dado não estruturado (anotação de reunião)

"Cliente João da Silva, fone 9999-8888, comprou 2 hambúrgueres no dia 10."

Dado estruturado (formato JSON)

{ "cliente": { "nome": "João da Silva", "telefone": "9999-8888" }, "pedido": { "data": "10/03/2026", "itens": ["hambúrguer", "hambúrguer"] } }

Seção 2.3

Identificação Inicial de Entidades — Teoria

A identificação inicial de entidades é o marco zero da modelagem de dados. Uma entidade representa um objeto, pessoa, conceito ou evento do mundo real sobre o qual o sistema precisa manter informações armazenadas.

Geralmente, as entidades são os "substantivos" principais que encontramos durante o levantamento conceitual.

❌ Não são entidades (ações)

  • Vender
  • Entregar
  • Comprar
  • Fabricar

✅ São entidades (agentes/objetos)

  • Vendedor
  • Cliente
  • Produto
  • Pedido
Resultado esperado
Identificar corretamente essas entidades garantirá que o Modelo Entidade-Relacionamento (MER) reflita com precisão o cenário do negócio da empresa contratante.

Streaming de Vídeo (Netflix, Amazon Prime)

Empresas de streaming lidam com petabytes de dados. Se eles não estruturarem a coleta de dados de forma rigorosa na identificação de entidades, seria impossível recomendar filmes com precisão ou continuar um episódio de onde você parou.

Loja de Departamentos (Magalu, Renner)

No e-commerce, a estruturação de dados define a logística. Se a entidade Pedido não for separada das entidades Nota Fiscal e Entrega, a empresa não consegue integrar o sistema com as transportadoras de forma eficiente.

FAQ — Dúvidas Reais de Alunos

Perguntas frequentes respondidas pelo professor.

Seção 5 · FAQ

Perguntas Frequentes (1/3)

1. Qual a diferença entre dado e informação no levantamento?
O dado é o fato bruto coletado (ex: "R$ 50,00", "05/03/2026"). A informação é o dado estruturado em um contexto útil (ex: "O valor da consulta no dia 05/03/2026 foi R$ 50,00").
2. Como sei se uma palavra é uma entidade ou um atributo?
Uma entidade é um objeto independente que possui características próprias (ex: Cliente). O atributo é a característica dessa entidade (ex: CPF do Cliente). Se a palavra apenas descreve outra coisa, é um atributo.
Seção 5 · FAQ

Perguntas Frequentes (2/3)

3. Eu preciso mapear todos os dados da empresa logo no início?
Não. Você deve focar nos dados que são estritamente necessários para solucionar os problemas de organização, controle e integração de informações definidos no escopo do projeto.
4. E se duas entidades parecerem a mesma coisa? (Ex: Usuário e Cliente)
Você precisa analisar as regras de negócio. Se o sistema trata Usuário (quem faz login) e Cliente (quem compra) com dados diferentes, podem ser entidades separadas. Caso contrário, unifique-as durante a sistematização.
Seção 5 · FAQ

Perguntas Frequentes (3/3)

5. A entrega parcial de hoje já deve ter o SGBD escolhido?
Não. A escolha e a justificativa para o SGBD dependerão das características dos dados que você está levantando hoje, mas o foco da Aula 4 é a conceituação.
